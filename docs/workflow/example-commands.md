# Claude Code 実行例

実際にどう依頼するかの具体例集。

---

## ES作成フロー

### 企業調査から始める
```
「[企業名]の企業調査をして」
```
→ `company-info/[企業名]/research_brief.md` が生成される

### ES草稿を作成
```
「[企業名]のESを書いて」
```
→ `移行後ES/[企業名].md` に回答が生成される

### ES全体をレビュー
```
「[企業名].mdを全体的にレビューして」
```
→ es-review-protocol が全エージェントを順番に呼び出す

### 特定の視点でレビュー
```
「[企業名].mdの設問に正面から答えているか厳しく確認して」
「[企業名].mdが[職種名]向けになっているか確認して」
「[企業名].mdで面接で突っ込まれそうな質問を出して」
```

### 改善
```
「設問3が設問ズレしているという指摘を踏まえて修正して」
「可読性の問題を修正して字数は350字以内に収めて」
```

## 企業調査フロー

### 調査の不足を確認
```
「[企業名]の企業調査の不足論点を確認して」
```

### 競合比較
```
「[企業A]と[企業B]と[企業C]を比較して志望動機に使えるポイントを整理して」
```

### 職種調査の確認
```
「[企業名]の[職種名]の職種調査に不足がないか確認して」
```

---

## 面接準備フロー

### 面接準備シートを作成
```
「[企業名]の[職種]コースの1次面接の準備をして」
```
→ `company-info/[企業名]/interview_prep_1次_YYYYMMDD.md` が生成される

### 深掘り質問を事前確認
```
「[企業名].mdで2次面接で詰められそうな質問を全部出して」
```

---

## エピソード管理フロー

### 新しいエピソードを追加
```
「[エピソードの概要]をエピソードライブラリに追加して」
```
→ episode-formatter が `ES/md/[番号]_[カテゴリ]_[タイトル].md` に保存

### 使えるエピソードを確認
```
「ES/md/INDEX.mdを見て、[テーマ]に使えるエピソードを教えて」
```

---

## 品質チェック（手動実行）

```bash
# ESファイルをチェック
python tools/checks/es_checker.py 移行後ES/[企業名].md

# 全企業の調査品質をチェック
python tools/checks/research_checker.py --all
```

## スキル更新フロー

### 面接フィードバック後
```
「今日の面接で[こういう指摘を受けた]。skill-update-logに記録して」
「[スキル名]に[この改善点]を反映して」
```

### ES改善内容を知識ベースに還元
```
「[企業名].mdの改善点をfoundationsに反映して」→ es-refiner
「es-writerスキルを今回の編集差分から改善して」→ es-improver
```

## 階層スキルを活用した高度なユースケース

### ケース1: AM企業×運用職のES最終仕上げ

AM企業のESを提出前に業種・職種の観点で最終確認したい場合。

```
「[ニッセイAM].mdを、AMの業種評価軸（フィデューシャリー・運用哲学）と
 運用職の評価軸（ポートフォリオ思考・長期コミット）で確認して」
```

→ company-fit-reviewer が `industries/asset-management-fit` + `roles/fund-manager-fit` を参照して評価

---

### ケース2: 競合3社の調査結果を比較して「なぜここか」を強化

複数社に応募中で、志望動機の差別化に行き詰まった場合。

```
「ニッセイAM・野村AM・大和AMの3社を比較して、
 なぜニッセイAMでないといけないかの論点を整理して」
```

→ `company-comparison-framework` + `research/asset-management-research-lens` を使って比較表を生成

---

### ケース3: 文系人事向けの表現チェック

IT知識のない人事担当者が読んでも理解できるかを確認したい場合。

```
「[企業名].mdを文系の新卒採用担当者（入社2年目）目線で確認して。
 専門用語が多い部分があれば言い換え案も出して」
```

→ readability-reviewer が `audiences/junior-humanities-readable-style` を参照して評価

---

### ケース4: 最終面接前夜の詰め込み対策

最終面接（役員面接）を翌日に控えた場合の集中対策。

```
「[企業名]の最終面接（役員面接）の対策をして。
 特に競合比較の反論と、なぜここかの深掘りに備えたい」
```

→ `stages/final-interview-depth-mode` + `interview/rebuttal-handling` + `interview/role-why-us-answer-patterns` を使って準備

---

### ケース5: ガクチカを1分・3分・5分バージョンで準備

面接の時間によって長さを変えたい場合。

```
「ES/md/001_*.mdのエピソードを使って、
 ガクチカの1分バージョンと3分バージョンを話し言葉で作って」
```

→ `interview/gakuchika-answer-patterns` が長さ別の構造テンプレートを提供

---

### ケース6: 職種調査のギャップ確認（銀行DS職）

銀行のデータサイエンス職に応募中で、職種理解が浅い気がする場合。

```
「[三菱UFJ銀行]のDS職について、職種調査に抜けがないか確認して。
 業務内容・必要スキル・キャリアパスの観点で」
```

→ role-research-reviewer が `research/data-science-research-lens` + `research/banking-research-lens` を参照してギャップを特定

---

### ケース7: ESの設問間重複を検出（3問以上のES）

複数の設問で同じエピソードや強みを使い回していないか確認したい場合。

```
「[企業名].mdの全設問でエピソードと強みの重複がないか確認して。
 重複マトリクスを作って」
```

→ consistency-overlap-reviewer + `overlap-detection` で重複マトリクスを生成

---

## スキル選択に迷ったら

```
「[企業名・職種・やりたいこと]に適したスキルを教えて」
```

→ `docs/workflow/skill-selection-matrix.md` を参照して最適なスキルの組み合わせを提案
