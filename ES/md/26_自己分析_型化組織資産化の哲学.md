# 自己分析・「型化→組織資産化」の一貫哲学

## 内容

### 1. 哲学の定義

「型化→組織資産化」とは、**個人の工夫を標準化（型化）し、組織全体が再現可能な資産として活用できる状態へ定着させる**という一貫したアプローチ。「一時回避（個人が気をつける）」ではなく「継続可能な仕組みの構築（誰でも再現できる型）」という行動原理が根底にある。

### 2. 各エピソードに貫かれた哲学

| エピソード | 型化した内容 | 組織への定着形態 |
|-----------|------------|----------------|
| 財務資料自動化（`03`） | 16時間の属人作業をPythonで自動化 | 変換マスタをYAML定義で外出しし、顧客自身が運用・修正できる設計 |
| 塾バイトリーダー（`05`） | 指導品質のばらつきを共通フォーマット化 | 報告書の必須項目定義→引き継ぎロスをゼロに近づける体制 |
| AI活用型化（`19`） | 数値合わせ手順をプロンプトとしてナレッジ化 | チーム全体が同品質の確認を行える標準化フォーマット |
| データ基盤整備（`20`） | メタデータ整備・データ定義の厳格化 | 全従業員が安心して使えるデータ基盤の構築 |
| OCアンケート企画（`21`） | 来場者ニーズ収集の仕組み化 | ガバナンス枠組みの設定→継続的なデータ活用基盤 |

### 3. 行動原理の本質

**思考の起点（価値観の原点）**：出発点は「自分が面倒だ・非効率だと感じることを減らしたい」というシンプルな感覚。自分が楽になりたい、自分が介在しなくても自動的に回ってほしいという動機から始まる。ただしそこで止まらず、「自分だけが楽になるより、チームや会社全体に広がった方が価値が大きい」と考えることで仕組み化へ向かう。

**価値観の到達点（全体最適）**：改善を進めるほど「個人最適ではなく全体最適」を重視するようになっている。「最大多数の最大幸福」に近い感覚であり、使われること自体が目的ではなく、使われた結果として組織全体の成果が上がることに価値を感じている。

**一時回避禁止の原則**：問題が発生したとき「自分が気をつけます」で終わらせず、「次の人も引っかからない構造を設計する」まで解決と定義する。これはFailure Controlの思想（`16_設計思想`）とも一致する。

**属人化は組織リスク**：担当者の退職や交代によって業務が停止するリスクを常に意識し、「自分がいなくても回る状態」を設計ゴールとする。財務資料自動化案件での経験（担当者退職による業務停止リスク）がこの意識の原点。

**「作ることより使われること」**：優れた設計ツールや分析結果があっても、現場に活かされなければ価値はゼロ。標準化フォーマット・プロンプトナレッジ・共通テンプレートという「誰でも使える形」での整備にこだわる。

**シンプル・拡張性・汎用性にこだわる理由**：仕組みを作る際にこの3点を重視するのは、そのような設計の方が運用しやすく・修正しやすく・他の場面にも広げやすく・より多くの人に使われやすい＝全体成果につながりやすいから。「便利さ」ではなく「広がりやすさ」が設計基準。

### 4. 面接での語り方

**質問例：「強みを教えてください」→「型化・仕組み化力」として語る骨格**

「私の強みは、個人の工夫を組織全体が使える仕組みに昇華させる『型化力』です。長期インターンでの財務資料自動化（16時間→30分）、塾バイトでの報告書フォーマット統一、チームへのAIプロンプトナレッジ共有など、複数の場面で一貫して『再現可能な仕組みの構築』を主導してきました。根底にあるのは『一時回避ではなく構造解決』という信念で、担当者が変わっても同品質が出せる状態を設計ゴールとしています。」

**質問例：「なぜこの職種を志望するのか」→ りそな専門コース向け語り方**

「一人が二十五人分の力を発揮できる組織の実現を、データの信頼性とITの活用から主導したいと考えます。長期インターンにおいて16時間を要した財務資料作成を三十分へ短縮した際、単なる自動化に留まらず、データ品質の抜本的改善と業務フローの再設計を主導しました。」
