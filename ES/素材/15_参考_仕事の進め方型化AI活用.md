# 仕事の進め方：型化・AI活用・再発防止ログ（参考資料）

## 内容

ESや面接で「仕事の進め方」「強みの証拠」を語る際の根拠・詳細として参照する資料。インターンでの失敗から構築した標準手順（型）をまとめたもの。

---

### 1. 方針（大枠）

- AIでコードを書く前に、AIと「処理手順・全体像」を相談し、設計を固めてから実装に入る
- 途中で要望追加が起きる前提で、工程見直し（設計の更新）とタスク整理の更新を「必ず」回す
- 中間データを出力し、検証・再実行コストを下げる（「簡単だが効く」手当ては最初から入れる）
- 検証は目視より差分抽出の自動化を優先し、意思決定者の時間と検証コストを守る

### 2. タスク整理フォーマット（受領直後に書く）

| 項目 | 内容 |
|------|------|
| 目的 | 何の意思決定/アウトプットのためか |
| 背景 | なぜ今それが必要か、誰の課題か |
| 期限 | 必達/希望/バッファの有無 |
| ステークホルダー | 依頼者/レビュー者/最終受け手/クライアント |
| インプットデータ | 所在、形式、期間、粒度、定義 |
| アウトプット | 形式、粒度、並び順、例外処理、期待値 |
| 大まかな工程 | 後述の設計に接続 |
| 不明点リスト | 受領時点で洗い出す |
| リスク/懸念 | 追加要望の可能性、データ品質、定義揺れ等 |
| 確認タイミング | どの工程に入る前に誰に何を確認するか |

### 3. 数値合わせ（突合）の型

1. エントリーデータ・明細データの前処理（キー整形、欠損/重複、型変換など）
2. 明細がエントリに全て含まれるかの確認（含まれない場合は原因調査へ）
3. 集計（商材/PJ/補助項目/月次 等、粒度を決める）
4. 両者をロング形式に揃える（比較しやすい形）
5. 差分抽出（どちらかにしか存在しない行、件数差分、絶対値閾値など）
6. クライアント共有用アウトプット（CSV + 口頭説明）

### 4. AI（Codex/LLM）活用の型（失敗からの学び）

- AIに書かせる前に、処理手順・検証観点・ログ項目（フィルタ条件等）を設計に書く
- AIに出す指示は「要件→入出力→手順→テスト（検証）→例外」の順で与える
- AIが生成したコードは、重要部分（フィルタ条件・集計キー・結合キー）を必ず人間がレビュー
- 違和感が出たら「勘で修正」ではなく、差分が出る最小単位（中間データ/チェック表）を作って原因特定する
- 中間データを早期に書き出し、繰り返し結合を避ける

### 5. 報連相（確認の機械化）

- タスク受領時点で全体を一旦確認し、不明点は早期に質問する（ボールを持たない）
- 工程ごとに「確認タイミング」をあらかじめ埋め込み、機械的に確認する
- 確認は私情を挟まず、タイミングで淡々と実行する
- 日次報告は「現在地（工程）」「次にやること」「詰まっている点」を型で送る

### 6. 差分の「見る基準」に関する考え方

- 意思決定者の時間を守るため、全件を深掘りせず「差が大きいもの」を優先して見る
- パーセンテージは母数が小さいと判断を誤るため採用しない。絶対差や分布で判断
- 閾値は固定ではなく、対象・分布により変わる（例：分布を見て上位5件以上という基準を設定）
