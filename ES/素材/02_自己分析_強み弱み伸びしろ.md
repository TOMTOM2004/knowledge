# 強み・弱み・伸びしろ（自己分析）

## 内容

### 1. 強み（自覚・実例ベース）

**型化・仕組み化：** 複雑な作業でも工程を外部化し、手順（型）に落として再現性を上げる。国語の読解改善や、数値合わせ/タスク整理のフォーマット化に表れている。

**改善志向：** 失敗をそのままにせず、フォーマットを作って更新し、再発防止につなげる。

**構造化・見立て：** 経済学の学習を通じて、構造的な関係や長期トレンドに目を向ける姿勢がある。

**検証力・実装力：** データ分析（集計ロジック作成→分析設計→レポート作成）を一連で担った経験がある。

**クライアントワーク適性（自己評価）：** 同一業務が繰り返されにくい環境で、タスクを抽象化し、汎用的な型を作れる。

**責任感：** 期限内にやり切れなかったときに強い悔しさが残り、改善の原動力になっている。

**独自のポジショニング「文理融合のブリッジ」：** 経済学のドメイン知識（文系）とPythonやデータ分析の実装力（理系）を兼ね備え、「作ることができる経済学部生」として理論を実装で検証できる。

**因果推論に基づく批判的思考：** 「データが一緒に動いている（相関）」だけで満足せず、「なぜそうなったのか（因果）」を突き止めるアカデミックな誠実さ。AIの出力結果や通説に対して「本当にそうか？」とデータで裏付けを取る姿勢。

**AIオーケストレーション：** 単一プロンプトを投げるのではなく、状況に応じて適切なAIの役割を呼び出す「メタ・スキル」を開発・運用。AIを「使う」のではなく「設計者（Architect）として指揮する」スタイルを確立。これにより1人では不可能な規模の仕組みを構築できる。

**失敗制御（Failure Control）：** 「失敗から学ぶ」という受動的な姿勢ではなく、失敗の根本原因（Root Cause）を特定し、再発防止のガードレールをシステムに組み込むことを重視。個人のミスで終わらせず、次の人も引っかからない構造設計で解決する。

**超高速技術習得（1年間の成長軌跡）：** 1年前はGoogle Colab専用・ローカル環境未経験の状態から、現在はCursor・Claude Code（CLI）を駆使したローカル開発環境を自力で設計・運用できるレベルへ到達。AIを活用した開発ドメイン知識の超高速補完を実証済み。

**中長期計画立案・逆算思考：** ゴールから逆算してマイルストーンを引く思考スタイルを強みとして自覚している。中学時代の文化祭でリーダーとして大規模プランを設計・実行した経験（中高全体2位の観客動員）が原点。業務上でもフロントローディング（着手前の論点洗い出し）として発揮されており、実装前に課題を構造化して手戻りを防ぐ動き方に一貫している。プライベートでも旅行計画の立案を担うことが多く、参加者の満足度が高い評価を継続して得ている（実績の積み上がり過程にある自覚はあるが、自己認識として明確な強みの一つ）。

### 2. 弱み・課題（自覚・反省点）

**設計の甘さ：** 先の要望（追加分析、期間変更等）を想定せずに実装を始め、手戻りやコード崩壊が起きた。

**AI活用の制御：** AIに書かせたコードを完全に理解せず、違和感ベースで修正し続けてしまう局面があった。原因特定ができず自信を失った。

**コミュニケーション（報連相）：** 進捗共有、確認のタイミング、アウトプット説明の端的さが課題として繰り返し出ている（克服意思は強い）。

**深掘り不足（器用貧乏）：** 広く浅く「それなりにできる」が、もう1〜2段深い技術・業務理解に踏み込めていない。

**飽き性：** 意味を感じない反復作業では集中が続きにくい。

**検証の自動化の遅れ：** 差分抽出など、機械的にできる検証を目視で行い時間を溶かす場面があった。

**デリゲーション不足：** 「自分でやった方が早い」という思考で抱え込み、他者の領域を潰してしまうことがある。目的/KPIが曖昧・分析手法が統一されない・納期リスクが見える時に出やすい。

**仕事を一人で抱え込む傾向：** 複数案件を並行処理する際に、自分が全タスクを把握・コントロールしようとするあまり、組織全体の負荷やスケジュールの可視性が下がってしまうことがある。インターンで複数クライアントの案件を同時に抱えた際、進捗の可視化と早期の相談タイミング設定が不十分で、上司への報告が遅れた経験がある。改善策として、（1）案件ごとの進捗をタスクリストで見える化する、（2）「詰まったら即相談」ではなく「〇〇の判断が必要になりそうなので事前に相談します」という先読み報告の癖づけ、を実践中。

### 3. 上司からのフィードバック（要約）

- 技術面はLLMにより差が出にくくなっている。だからこそ現代に求められるのは報連相、期日遵守
- 反省からフォーマットを作成し更新する姿勢は良い
- 分析結果が揺れる事態を防ぐため、設計を先に固め、フィルタ条件等をログとして設計に明記することが重要
- 仕事ができる人は、受領時点でタスク全体を確認し、不明点は早期に質問して「ボールを自分が持ち続けない」進め方をする
- 「技術は高いが報告が遅い人」より「技術はそこそこでも、遅れそうな点を先に把握しこまめに報告する人」と仕事をしたい

### 4. 「器用貧乏」の言い換え候補（強みに寄せる）

- 吸収が速く、未知のタスクでも短期間で立ち上がれる（立ち上げ力 / ラーニングアジリティ）
- 状況に応じて工程を設計し直し、最適な手段（自動化/可視化/型化）を選べる（適応力）
- ただし「専門性の芯」を作るために、ドメイン知識の深掘りと検証設計の精度を上げる段階にある（伸びしろの提示）

### 5. 伸びしろ（次に伸ばすべき領域）

1. 受領時のタスク整理→設計→実装→検証→報告の一連を、常に同じ品質で回す（「機械的」に回せる状態へ）
2. フィルタ条件・定義・前提をログ化し、成果物の再現性と説明可能性を担保する
3. 数値合わせ/検証の自動化（差分抽出、例外の特定、アラート設計）を先に作る癖をつける
4. ドメイン知識を深める（資産運用、企業価値評価、M&A/資本政策、マクロと企業業績の接続など）
5. 自分の「目標（将来像）」を定義し、やらないことを選べるようにする（分散の抑制）

### 6. モチベーションの源泉（3つのハイブリッドエンジン）

**① 先回りする危機感（攻撃型のオフェンス）：** AIの劇的な進化による仕事代替の脅威や、理系人材との技術的ギャップに対し、常に強い危機感を抱いている。これを能動的・野心的な生存戦略として昇華。

**② カバーの精神（他者貢献と仕組み化）：** 中学時代のバスケットボールで培った「仲間が抜かれたら自分が補う」精神。現場で皆がミスをして困っている複雑な業務を察知し、AIやシステムを用いてフォーマット化・解決策を設計する利他的な行動原理。

**③ 知的好奇心と「具体・抽象の往復」の快感：** バラバラの事象（具体）が、自らの思考によって一本の線（抽象・型・モジュール）に繋がった瞬間に強いやりがいを感じる。

## 面接・ES活用メモ

- 「弱みを教えてください」→ 器用貧乏（ラーニングアジリティとして言い換え）、報連相の改善プロセスを具体的に語る。または「仕事を一人で抱え込む傾向」を弱みとして正直に語り、改善策（進捗の可視化・先読み報告）とセットで提示する
