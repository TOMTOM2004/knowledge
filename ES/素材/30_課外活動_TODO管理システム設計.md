# Claude Code の全プロジェクト横断 TODO 管理システムの設計・実装

## 概要

9個の TODO ファイルが分散し別セッションから着手状況が不可視という課題に対し、スキャンスクリプト + ステータス管理スキル + SessionStart hook の3層構成で全プロジェクト横断の TODO 一元管理システムを設計・実装。1セッション（約2時間）で設計から実装・テスト・運用ルール整備まで完了し、5プロジェクト計110件超のタスクを一目で把握可能にした。

## STAR形式

### Situation（状況）

個人で複数のプロジェクト（卒業論文・就活ナレッジベース・個人開発ツール・会社業務）を並行運用しており、各プロジェクトの TODO を Markdown ファイル（`tasks/todo.md`）で管理していた。しかし、9個の TODO ファイルが個人プロジェクト（ローカル）・個人開発リポジトリ・会社 Dropbox の3つのゾーンに分散しており、以下の問題が発生していた:
- あるセッションでタスクに着手しても、別セッションからはその着手状況が見えない
- 完了項目が todo.md に残り続け、未着手タスクが埋もれて全体像が掴めない
- TODO ファイルのフォーマットがプロジェクトごとに異なる（handoff テンプレート形式 / checkbox 形式）

### Task（役割・目標）

全プロジェクトの TODO を横断的に把握でき、かつタスクの着手状態が別セッションからも可視化される仕組みを構築する。具体的には:
- 全 TODO ファイルをスキャンしてグローバルインデックスを自動生成する
- タスクのステータスライフサイクル（未着手 → 着手中 → 完了）を管理する
- セッション開始時に自動で全体状況を表示する
- 会社情報のセキュリティを担保しつつ横断管理する

### Action（行動）

1. **課題の構造化**: まずサブエージェントを使って全 TODO ファイルの分布・フォーマット・件数を網羅的に調査。9ファイル・2種フォーマット・3ゾーンという構造を可視化した
2. **設計判断**: 「静的インデックス生成」と「動的クエリ」のハイブリッド方式を選定。Plan モードで詳細設計を行い、ユーザー承認を得てから実装に着手した
3. **スキャンスクリプトの実装**（Python）:
   - 2種のパーサー（handoff テンプレート / checkbox）を実装し、ファイル内容からフォーマットを自動判定
   - `auto_discover` 機能で新規プロジェクトの TODO ファイルを自動検出
   - 会社情報は `abstraction_filter`（既存の機密情報置換モジュール）を適用してインデックス上で抽象化
   - work zone のプロジェクトラベルを `work-N` に匿名化（多層防御）
4. **ステータス管理スキルの設計**: 着手（Mode A）/ 完了（Mode B）/ 状況確認（Mode C）の3モードを持つ Claude Code スキルを作成。着手時に `_started:` タイムスタンプを付与し、24時間超の着手中項目を stale フラグで警告する仕組みを導入
5. **テスト駆動の品質改善**: `--dry-run` で実データをパースし、ノイズ（「なし」「（なし）」の誤検出）やセクション境界の誤判定を発見 → 即修正を3回繰り返した
6. **運用ルールの整備**: グローバル設定ファイル（CLAUDE.md）のワークフローに「着手宣言」「完了宣言」ステップを追加。done 管理ルール（月別ファイルへの退避）を全プロジェクト共通化
7. **既存データの棚卸し**: 全プロジェクトの着手中項目を精査し、完了項目53件を done/ に一括移動。todo.md を未着手・着手中のみのクリーンな状態に整理

### Result（結果・学び）

- 5プロジェクト横断で**未着手47件・着手中1件・完了63件**を一目で把握可能になった
- セッション開始時に自動で全体状況が表示され、**着手状態が別セッションから可視化**されるようになった
- **設計から実装・テスト・運用ルール整備まで1セッション（約2時間）で完了**
- 学び: 実データでのテスト駆動が重要（机上設計では気づかないパーサーのエッジケースが3件発見された）。また、セキュリティ（会社情報の抽象化）と利便性（横断管理）の両立には、多層防御（匿名ラベル + abstraction_filter + git 管理外配置）が有効だった

## 生の記録（草案原文）

### タイトル
Claude Code の全プロジェクト横断 TODO 管理システムの設計・実装

### 概要
複数プロジェクト（卒業論文・就活ナレッジベース・claude-brain・会社 Dropbox）にまたがる TODO が分散し、別セッションから着手状況が見えないという課題を発見。スキャンスクリプト + ステータス管理スキル + SessionStart hook の3層構成で、全プロジェクトの TODO を一元把握・ステータス同期する仕組みを1セッションで設計から実装・テスト・運用ルール整備まで完了した。

### 背景・課題
- 9個の todo.md が個人プロジェクト、claude-brain、会社 Dropbox に分散
- あるセッションでタスクに着手しても、別セッションからはその状況が分からない
- 完了項目が todo.md に残り続け、未着手タスクが埋もれる

### 取り組み内容
1. **課題分析**: サブエージェントで全 todo.md の分布・フォーマット・件数を調査
2. **設計**: A（静的インデックス）+ B（動的クエリ）のハイブリッド方式を選定。Plan モードで詳細設計
3. **実装**:
   - `todo_scanner.py`（Python）: 2種のパーサー（handoff テンプレート / checkbox）でフォーマット自動判定、auto_discover で新規プロジェクト自動検出、`abstraction_filter` で会社情報を抽象化
   - `todo-status-manager` スキル: 着手（Mode A）/ 完了（Mode B）/ 状況確認（Mode C）の3モード
   - SessionStart hook: セッション開始時に自動スキャン・サマリー表示
4. **ステータスライフサイクル設計**: 未着手 → 着手中（_started: タイムスタンプ付与）→ 完了（done/ に月別退避）。24h 超の着手中は stale フラグ
5. **zone 分離**: Local / Dropbox でインデックスをセクション分け。セキュリティ上、インデックスは git 管理外（`~/.claude/state/`）に配置
6. **既存 TODO の棚卸し**: 全プロジェクトの着手中項目を精査、完了項目を done/ に一括移動
7. **運用ルール整備**: CLAUDE.md のワークフローに「着手宣言」「完了宣言」ステップを追加、done 管理を全プロジェクト共通化

### 工夫・こだわり
- パーサーのノイズフィルタ（「なし」「（なし）」等を除外）やセクション境界検出（`###` 見出しで State snapshot パース終了）など、実データで発見した問題を即修正するテスト駆動のアプローチ
- work zone のラベルを `work-N` に匿名化し、`abstraction_filter` と組み合わせてセキュリティを多層防御
- 冪等な再生成（ソースから毎回生成）で同時セッション問題を回避

### 成果
- 5プロジェクト横断で未着手47件・着手中1件・完了63件を一目で把握可能に
- セッション開始時に自動で全体状況が表示される
- 着手状態が別セッションから可視化され、二重着手や状況不明を防止

### 使用技術・ツール
Python, Claude Code（hooks / skills / subagents）, abstraction_filter, Markdown パーサー
