# ES Improver スキル

es-writer 初稿（main）とユーザー編集版（ブランチ）の差分を分析し、
es-writer/SKILL.md を自動更新してESライターの品質を継続的に向上させる。

---

## Step 0: 入力確認

ユーザーの依頼から **企業名** を特定する（省略時は差分があるファイルから自動判定）。

リポジトリパス: `/Users/ishidatomonori/Desktop/knowledge`

---

## Step 1: ブランチ自動検出

```bash
git -C /Users/ishidatomonori/Desktop/knowledge branch --list \
  | grep -v '^\*\? *main$' \
  | grep -v '^\*\? *master$' \
  | sed 's/^[* ]*//'
```

| 結果 | 対応 |
|------|------|
| 0件 | 「ユーザー編集ブランチが見つかりません。mainブランチ以外のブランチを作成してコミットしてください」と報告して終了 |
| 1件 | 自動選択して次へ |
| 2件以上 | ブランチ一覧を提示してユーザーに選択を求める |

選択したブランチを `<edit-branch>` とする。

---

## Step 2: 差分の対象ファイル特定

```bash
# 差分があるESファイルを列挙
git -C /Users/ishidatomonori/Desktop/knowledge diff main..<edit-branch> \
  --name-only -- "移行後ES/"
```

企業名が指定されていた場合は該当ファイルのみを対象とする。
指定がない場合は差分ファイル全件を対象とする。

---

## Step 3: 差分を取得

対象ファイルごとに以下を実行する。

```bash
git -C /Users/ishidatomonori/Desktop/knowledge diff main..<edit-branch> \
  -- "移行後ES/<企業名>.md"
```

差分から以下を分類する:

| 記号 | 意味 |
|------|------|
| `+` 行 | ユーザーが追加・変更した内容 |
| `-` 行 | es-writer の初稿にあったがユーザーが削除・変更した内容 |

---

## Step 4: 不足分析

差分の内容を以下の観点で分析し、**es-writer が出力できなかった理由**を推定する。

### 4-1. 変更カテゴリの分類

各 `+` 行（ユーザー追加）を以下のカテゴリに分類する:

| カテゴリ | 判定基準 | 対応するSKILL.mdセクション |
|---------|---------|--------------------------|
| **企業固有情報の追加** | 会社名・サービス名・戦略・数値・部門名などが追加 | Step 3（ナレッジ収集）の research_brief.md 参照強化 |
| **表現・語彙の変更** | 単語・言い回しが変わった（意味は同じ） | G2/G3 ガードレールに新パターン追加 |
| **構成・順序の変更** | 段落の並び替え・PREP法の修正など | Step 4（設問種別マッピング）に構成ルール追加 |
| **内容の追加（設問固有）** | 設問に固有の視点・経験が追加 | Step 4-2（回答生成ルール）に追記 |
| **視点の追加（汎用）** | 複数企業に共通して使える観点・フレームが追加 | G1〜G8 または新ガードレールとして追加 |
| **削除・簡略化** | es-writer の冗長な表現がユーザーに削除された | 回答生成ルールに「冗長表現の禁止パターン」追加 |

### 4-2. 根本原因の推定

各カテゴリについて「なぜes-writerが出力できなかったか」を以下の軸で推定する:

- **情報不足**: research_brief.md の参照が不十分 / 参照していない情報源がある
- **ルール不足**: SKILL.md に該当するガードレール・生成ルールが存在しない
- **ルールの曖昧さ**: ルールはあるが具体例・禁止パターンが足りない
- **エピソード選定ミス**: research_brief.md の適合マッピングに沿っていない

---

## Step 5: 改善提案の生成

以下の形式で改善提案をユーザーに提示する。**この時点ではまだSKILL.mdを更新しない。**

```
## ES Improver 分析結果: <企業名>

対象ブランチ: <edit-branch>
差分ファイル: 移行後ES/<企業名>.md

---

### 検出した不足パターン

#### [1] <カテゴリ名>
- **変更内容**: （ユーザーが何を変えたか、diff の +/- 行を引用）
- **根本原因**: （なぜes-writerが出力できなかったか）
- **提案する更新箇所**: SKILL.md の <セクション名>
- **追加・修正内容（案）**:
  ```
  （SKILL.mdへの具体的な追記テキスト）
  ```

#### [2] ...

---

### SKILL.md 更新サマリー

| No. | 更新箇所 | 変更種別 | 1行サマリー |
|----|---------|---------|-----------|
| 1 | Step X / G# | 追加/修正/削除 | ... |

---

更新を適用しますか？（「適用して」と言っていただければ es-writer/SKILL.md を更新します）
```

---

## Step 6: ユーザー承認後に SKILL.md を更新

ユーザーが「適用して」「OK」「はい」などと承認したら:

1. `/Users/ishidatomonori/.claude/skills/es-writer/SKILL.md` を Read で読み込む
2. 提案した変更を Edit で適用する
3. 変更の概要を報告する:

```
✅ es-writer/SKILL.md を更新しました。

更新箇所:
- <セクション名>: <変更内容の1行サマリー>
- ...

次回から es-writer はこれらの改善を反映して ES を生成します。
```

部分的に却下された提案は適用しない。

---

## Step 7: フィードバックループ記録（任意）

更新内容を `mcp__memory__save_lesson` に記録して将来の参照に備える。

記録フォーマット:
```
タイトル: es-writer改善 - <企業名>差分から検出した不足パターン
内容: <カテゴリ>: <根本原因> → <SKILL.mdの更新箇所>
```

---

## 注意事項

- SKILL.md の更新は **追記・修正のみ**。既存のガードレールや生成ルールを削除する場合はユーザーに明示的に確認する
- 企業固有の情報（会社名・数値・固有戦略）は SKILL.md に書かない（汎用ルールのみ抽出する）
- 差分が大量の場合（100行超）は「企業固有情報の追加」カテゴリを優先し、汎用パターンのみ提案する
- `ES/submitted/` のファイルは読み取り専用（書き込み禁止）
