# Phase 2: Investigation Loop（調査ループ）

**目的: ✗を可能な限りゼロに、△を可能な限り◎に引き上げる。**

`references/web-query-templates.md` を読み込む。

## ループ構造

```
Round 1: 業界横断データ + 一次ソース直接取得
  ↓ 進捗レポート → 終了条件判定
Round 2: 二次ソース + WebSearch補完
  ↓ 進捗レポート → 終了条件判定
Round 3: クロス軸再評価 + 三次ソース + フォールバック
  ↓ 進捗レポート → 終了条件判定
Round N: 残存セルへの追加調査（必要な場合のみ）
  ↓ 進捗レポート → 終了条件判定
```

## 終了条件

以下のいずれかを満たしたらループを終了し Phase 3 へ進む:

| 条件 | 判定 |
|------|------|
| 全セルが ◎ or 原理的制約（`✗[非公開]` / `△[定性のみ]`） | **即終了** |
| 2ラウンド連続で◎化がゼロ | **終了**（追加調査しても改善しない） |
| 最優先軸の `✗[未調査]` がゼロ、残存△が原理的制約のみ | **終了可能**（ユーザーに確認） |

### 1-c. HTMLページ取得方法

WebFetch で 403 エラーが発生した場合、Playwright MCPで実ブラウザ経由でアクセスする:

1. `mcp__playwright__browser_navigate` でURLを開く（ドメインは `www.` 付きを使用。例: `www.myam.co.jp`）
2. `mcp__playwright__browser_snapshot` でページ内容をアクセシビリティツリーとして取得
3. 取得したデータには出典に `[Playwright MCP経由]` タグを付ける

403にならない場合は通常の WebFetch を使用する。

### 1-d. PDF取得方法

**重要: Playwright MCPのsnapshotではPDFの中身は取得できない（空になる）。PDFは必ず以下の手順で取得する:**

1. `Bash: curl -sL -o /tmp/<filename>.pdf "<URL>" -w "%{http_code}"` でダウンロード
2. `Read(file_path="/tmp/<filename>.pdf", pages="1-5")` で画像として読み取る（1回最大20ページ）
3. 大きいPDFは目次ページを先に読み、必要なページ範囲を特定してから分割読み取り

PDFのURLが不明な場合の探し方:
- **目論見書**: 各AM会社の `/fund/unyou_report.json`（存在する場合）にPDFリンク一覧がJSON形式で格納されている
- **スチュワードシップ報告書**: スチュワードシップ・コード対応ページにPDFリンクが掲載されている
- **財務届出書**: 財務状況等ページにPDFリンクが掲載されている

取得したデータには出典に `[curl+Read]` タグを付ける。

## Round 3: クロス軸再評価 + 三次ソース + フォールバック

### 3-a. クロス軸再評価

**単一ソースでは△でも、複数ソースを組み合わせると◎に引き上げられるケースが多い。**

残存△セルについて以下を確認する:

1. **別軸のデータ流用**: 別の軸で取得したデータがこの軸にも使えないか？
   - 例: 財務届出書のPLデータ → A-9 Fee resilience の裏付け
   - 例: スチュワードシップ報告書の対話件数 → B-5 Time horizon の間接証拠
2. **定性+定量の組み合わせ**: 定性情報と定量情報を合わせると◎基準を満たさないか？
   - 例: A-4 相互会社の事実 + 非上場＝短期圧力排除 + エンゲージメント長期時間軸 → 構造分析として◎
3. **transcript_findings の活用**: `company-info/<企業名>/transcript_findings_*.md` が存在する場合、面談会で得た一次情報で補強できないか？

◎化可能なセルは記述を充実させて◎に引き上げる。

### 3-b. 三次ソース調査

Round 2 のデータソース優先順位表の「三次ソース」列を調査する。

### 3-c. フォールバック（残存✗への最終手段）

1. **WebSearch スニペットからのデータ抽出**: 検索結果のスニペットに数値や事実が含まれている場合はそれを使用する
2. **親会社・グループ会社の公開資料**: 子会社サイトがブロックされていても親会社からグループ全体の情報を取得できる場合がある
3. **面談会transcript_findings**: 公式資料にない定性情報（現場の雰囲気、意思決定の実態、キャリアパスの実情等）
4. **手動補完依頼**: 上記すべてで取得できない場合、ユーザーにURL一覧を提示し「ブラウザで開いてテキストを貼ってください」と依頼する

フォールバックで取得したデータには出典に `[フォールバック: <手段>]` タグを付ける。

### 3-d. 原理的制約の確定

以下の軸は公開情報からは◎化が困難。タグを付けて確定する:

| 軸 | 理由 | 確定タグ |
|---|---|---|
| A-3 グループ内役割分担 | 本社運用部門との棲み分けは内部情報 | `△[定性のみ]` or `✗[非公開]` |
| A-5 パッシブ/アクティブ比率 | 数値はディスクロージャー誌 or 有報でのみ取得可能 | `△[定性のみ]` |
| B-1a Information edge | アナリスト人数・調査投資額は業界慣行として非公開 | `△[定性のみ]` |
| B-1b Alternative data | 非伝統的データ活用。使っていない企業は✗が正しい評価 | `✗[非公開]` |
| B-4 Portfolio construction（定量） | Active Share等は運用報告書（全体版）でのみ取得可能 | `△[定性のみ]` |
| B-5 Time horizon（定量） | ターンオーバー率は運用報告書（全体版）でのみ取得可能 | `△[定性のみ]` |
| B-6 Risk management | 損切り規律・下落局面対応は内部運用ルール | `✗[非公開]` |

# Phase 3: 比較表構築 + Insight + 品質チェック

Investigation Loop 終了後（＝原理的限界まで情報を集め切った後）に実行する。

## 3-1. 横断比較表の構築

`references/comparison-template.md` を読み込み、テンプレートに沿って比較表を生成する。

### セル記述ルール

- **2-3行以内**: 事実データ + 1ワード評価（強/中/弱/不明）
- **最優先軸**: 必ず出典（URL or 資料名）を記載
- **数値必須軸**（A-5, A-8, D-1）: 定量データを含める。定性記述のみはNG
- **差別化フィルタ**: 4社以上に同じ記述が当てはまるセルは書き直す
- **原理的制約セル**: タグと理由を比較表内に記載（例: `△[定性のみ] アナリスト人数非公開、定性的にはセクター制を確認`）

### 3層モデルの意識

am-comparison-framework.md の3層モデルに従い、各セルがどの層の情報かを意識する:

| 層 | 内容 | 対応する群 |
|---|------|-----------|
| 1. 表の説明層 | 会社が何を言っているか | 全群（1層だけにならないよう注意） |
| 2. 運用プロセス層 | 実際にどう意思決定するか | 主にB群 |
| 3. 組織・事業モデル層 | 継続的に回せるか | 主にA群・D群 |

## 3-2. Insight Layer 生成

### 軸別リーダー/ラガー

各軸でどの企業がリードし、どの企業がラグかを1行で記述。

### 構造パターン（母体別）

| パターン | 該当企業 | 共通特徴 |
|---------|---------|---------|
| 生保系 | MYAM, ニッセイAM | ALM背景、長期投資志向、安定受託基盤 |
| 証券系 | 野村AM, 大和AM | リテール販売力、商品開発力 |
| 複合母体 | AM-One, 三井住友DSAM | 複数母体のシナジーと制約 |
| 損保系 | 東京海上AM | 損保特有の顧客基盤 |
| 農協系 | 農林中金AM | 農林水産系資金の特殊性 |

### MYAMの相対的ポジション

最優先軸すべてで MYAM の位置づけを記述する。

### 「MYAMにしか言えないこと」抽出

比較表から「他社にはない、MYAMだけに当てはまる事実」を最低3項目抽出する。
各項目は対応する軸番号と紐づける。

→ この抽出結果は A-04（MYAM志望動機の固有性確定）で直接使用される。

## 3-3. 品質チェック

`references/quality-check-am.md` を読み込み、品質検証プロセスを実行する。

**Investigation Loop で既にセル品質は管理済みだが、以下の観点で最終確認する:**

| チェック | 基準 |
|---------|------|
| 出典検証 | 最優先軸の全◎セルにURL or 資料名が記載されている |
| 数値根拠 | A-5, A-8, D-1 に定量データが入っている |
| 差別化フィルタ | 4社以上に同じ記述が当てはまるセルがないか |
| MYAM完全性 | MYAM列の最優先軸に `✗[未調査]` がゼロ |
| 「立派に見える言葉」フィルタ | 「ESGに取り組んでいる」等の汎用記述がないか |
| 3層モデル検証 | 1層のみのセルが多すぎないか |
| transcript照合 | MYAM列が面談会での発言と矛盾していないか |

品質チェックで不足が見つかった場合は、Phase 2 に戻って追加調査する。

## 3-4. 最終ギャップマトリクス

比較表とともに最終的なギャップマトリクスを提示する:
