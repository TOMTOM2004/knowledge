# AM Competitor Researcher スキル

複数のAM企業を4群28軸で横断比較し、構造的な差別化ポイントを可視化する。

**正規の軸定義**: `docs/am-comparison-framework.md`（canonical source。スキル側で複製しない）

**参照ファイル（必要時に読み込む）:**
- `docs/am-comparison-framework.md` — 4群28軸の定義・3層モデル・比較原則（Step 0で必ず読み込む）
- `references/web-query-templates.md` — 軸ごとのWeb調査クエリ（Step 3で読み込む）
- `references/comparison-template.md` — 横断比較表のスケルトン（Step 4で読み込む）
- `references/quality-check-am.md` — 品質チェックプロセス（Step 7で読み込む）

## Step 1: 既存 research_brief 読み込み（並列）

対象企業ごとに以下を並列で読み込む。

```
Read: /Users/ishidatomonori/Desktop/knowledge/company-info/<企業名>/research_brief.md
```

### 1-1. データ抽出

各 research_brief から28軸に対応するデータを抽出し、以下のいずれかに分類する:

| 評価 | 定義 |
|------|------|
| ◎ | 具体的な事実データ・数値がある（出典付き） |
| △ | 言及はあるが具体性に欠ける（「ESG対応あり」等） |
| ✗ | 記載なし |

### 1-2. research_brief 未作成の企業

research_brief.md が存在しない企業を検出した場合:

```
「以下の企業は research_brief.md が未作成です:
  - <企業名>
  
先に company-researcher で research_brief を作成することを推奨します。
research_brief なしで Web 調査のみで比較を進めますか？」
```

ユーザーが続行を選択した場合、当該企業は Step 3 の Web 調査のみで埋める。

## Step 3: Web補完調査（軸別・並列）

`references/web-query-templates.md` を読み込み、✗・△セルを埋める。

### 3-1. 調査順序

1. **最優先軸の✗セル**（最初に全て埋める）
2. **最優先軸の△セル**（具体性を補強）
3. **次点軸**（優先度フィルタが `最優先+次点` 以上の場合）
4. **補助軸**（優先度フィルタが `全28軸` の場合）

### 3-2. 調査の実行

軸ごとに WebSearch / WebFetch を並列実行する。

- 1軸あたり最大3クエリ
- 企業ごとではなく**軸ごとにまとめて調査**する（同じ軸を全社分一度に比較できるため）
- スチュワードシップ報告書（C群）は各社の公式サイトから PDF を取得

### 3-3. HTMLページ取得

WebFetch で 403 エラーが発生した場合、Playwright MCPで実ブラウザ経由でアクセスする:

1. `mcp__playwright__browser_navigate` でURLを開く（ドメインは `www.` 付きを使用。例: `www.myam.co.jp`）
2. `mcp__playwright__browser_snapshot` でページ内容をアクセシビリティツリーとして取得
3. 取得したデータには出典に `[Playwright MCP経由]` タグを付ける

403にならない場合は通常の WebFetch を使用する。

### 3-4. PDF取得

**重要: Playwright MCPのsnapshotではPDFの中身は取得できない（空になる）。PDFは必ず以下の手順で取得する:**

1. `Bash: curl -sL -o /tmp/<filename>.pdf "<URL>" -w "%{http_code}"` でダウンロード
2. `Read(file_path="/tmp/<filename>.pdf", pages="1-5")` で画像として読み取る（1回最大20ページ）
3. 大きいPDFは目次ページを先に読み、必要なページ範囲を特定してから分割読み取り

PDFのURLが不明な場合の探し方:
- **目論見書**: 各AM会社の `/fund/unyou_report.json`（存在する場合）にPDFリンク一覧がJSON形式で格納されている
- **スチュワードシップ報告書**: スチュワードシップ・コード対応ページにPDFリンクが掲載されている
- **財務届出書**: 財務状況等ページにPDFリンクが掲載されている

取得したデータには出典に `[curl+Read]` タグを付ける。

### 3-5. フォールバック

上記で取得できない場合:

1. **WebSearch スニペットからのデータ抽出**: 検索結果のスニペットに数値や事実が含まれている場合はそれを使用する
2. **親会社・グループ会社の公開資料**: 子会社サイトがブロックされていても親会社からグループ全体の情報を取得できる場合がある
3. **面談会transcript_findings**: `company-info/<企業名>/transcript_findings_*.md` が存在する場合、面談会で得た一次情報を活用する。公式資料にない定性情報（現場の雰囲気、意思決定の実態、キャリアパスの実情等）が含まれている
4. **手動補完依頼**: 上記すべてで取得できない場合、ユーザーにURL一覧を提示し「ブラウザで開いてテキストを貼ってください」と依頼する

フォールバックで取得したデータには出典に `[フォールバック: <手段>]` タグを付ける。

## Step 4: 横断比較表の構築

`references/comparison-template.md` を読み込み、テンプレートに沿って比較表を生成する。

### 4-1. セル記述ルール

- **2-3行以内**: 事実データ + 1ワード評価（強/中/弱/不明）
- **最優先軸**: 必ず出典（URL or 資料名）を記載
- **数値必須軸**（A-5, A-8, D-1）: 定量データを含める。定性記述のみはNG
- **差別化フィルタ**: 4社以上に同じ記述が当てはまるセルは書き直す

### 4-2. 3層モデルの意識

am-comparison-framework.md の3層モデルに従い、各セルがどの層の情報かを意識する:

| 層 | 内容 | 対応する群 |
|---|------|-----------|
| 1. 表の説明層 | 会社が何を言っているか | 全群（1層だけにならないよう注意） |
| 2. 運用プロセス層 | 実際にどう意思決定するか | 主にB群 |
| 3. 組織・事業モデル層 | 継続的に回せるか | 主にA群・D群 |

## Step 8: △軸再評価

品質チェック後、△と評価された軸を再評価する。**単一ソースでは△でも、複数ソースを組み合わせると◎に引き上げられるケースが多い。**

### 8-1. 再評価の手順

1. △軸を一覧化する
2. 各軸について、以下の観点で既に手元にあるデータを再確認する:
   - 別の軸で取得したデータがこの軸にも使えないか？（例: 財務届出書のPLデータ → A-9 Fee resilience の裏付けに使える）
   - 定性情報と定量情報を組み合わせると◎基準を満たさないか？（例: A-4 相互会社の事実 + 非上場＝短期圧力排除 + エンゲージメント長期時間軸 → 構造分析として◎）
   - transcript_findings の一次情報が補強に使えないか？
3. ◎化可能な軸は記述を充実させて◎に引き上げる
4. 原理的制約で◎化不可能な軸は理由を明記して△のまま確定する

### 8-2. 原理的制約リスト

以下の軸は公開情報からは◎化が困難。△のまま理由を付記して確定する:

| 軸 | 理由 |
|---|---|
| A-3 グループ内役割分担 | 本社運用部門との棲み分けは内部情報 |
| A-5 パッシブ/アクティブ比率 | 数値はディスクロージャー誌（存在しないAMもある）or 有価証券報告書でのみ取得可能。定性的にはアクティブ中心を確認するにとどまる |
| B-1a Information edge | アナリスト人数・調査投資額は業界慣行として非公開。面談会一次情報で定性的な情報優位は確認可能 |
| B-1b Alternative data | 衛星画像・POS等の非伝統的データ活用。使っていない企業は✗が正しい評価 |
| B-4 Portfolio construction（定量） | Active Share・組入銘柄数・ターンオーバー率は運用報告書（全体版）でのみ取得可能。交付目論見書には未記載 |
| B-5 Time horizon（定量） | ターンオーバー率は運用報告書（全体版）でのみ取得可能 |
| B-6 Risk management | 損切り規律・下落局面対応は内部運用ルールであり非公開 |

### 8-3. 再評価レポート

```
## △軸再評価結果

### ◎に引き上げた軸
| 軸 | 元の評価理由 | 再評価根拠 |
|---|-----------|---------|

### △のまま確定した軸
| 軸 | 理由（原理的制約 or 追加調査で解消見込み） |
|---|--------------------------------------|

### 次のアクション
- [EDINET等で追加取得可能な軸の調査提案]
```
