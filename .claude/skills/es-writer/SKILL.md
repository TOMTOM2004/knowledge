# ES Writer スキル

ユーザーから企業名・設問情報を受け取り、ナレッジベースを参照してESを生成し、
プレビューを提示したうえで `移行後ES/<企業名>.md` に上書き保存する。

**参照ファイル（必要時に読み込む）:**
- `references/guardrails.md` — G1〜G11 品質チェックルール（Step 4完了後に必ず読み込む）
- `references/multi-question-design.md` — 3問以上ある場合の設計ロジック（Step 3後に判定して読み込む）
- `references/audience-styles.md` — 読者別の表現スタイル（人事/文系若手/現場実務者向け。Step 3で文体調整時に読み込む）
- `references/question-type-patterns.md` — 設問タイプ別の構造・配分パターン（志望動機/ガクチカ/強み弱み/ビジョン。Step 3で設問タイプ判定後に読み込む）

- **企業名**: 「〇〇のES」「〇〇の設問」などから抽出
- **移行後ESファイル**: `/Users/ishidatomonori/Desktop/knowledge/移行後ES/<企業名>.md`

ファイルが存在しない場合は「`移行後ES/<企業名>.md` が見つかりません。ファイル名を確認してください」と伝え、`ls /Users/ishidatomonori/Desktop/knowledge/移行後ES/` の結果を提示して終了。

## Step 4: 設問ごとにES回答を生成

### 4-1. 設問種別マッピング

| 設問に含まれるキーワード | 種別 |
|------------------------|------|
| 学生時代・力を入れた・頑張った・ガクチカ・努力 | ガクチカ |
| 志望・志望動機・入社したい理由・なぜ当社 | 志望動機 |
| 自己PR・強み・長所・自分の特徴 | 自己PR |
| チームワーク・協力・組織 | チームワーク |
| 失敗・挫折・困難・壁 | 失敗談 |
| 弱み・短所・改善 | 弱み |
| 将来・キャリア・10年後・ビジョン・やってみたいこと | ビジョン |
| 趣味・特技・好きなこと | 趣味特技（本人らしさ優先） |
| 資格・語学 | 資格（既存情報転記） |

### 4-2. 回答生成ルール

1. `best_answers.md` の該当設問のベストアンサーを起点とする
2. 企業名・敬称（Step 2）を差し込む
3. 志望動機は企業固有の情報（事業・ビジョン・戦略）を必ず入れる
   - 既存の `motivation.md` の同業種の記述を参考にしつつ、企業名に合わせて調整
   - 企業固有情報が不明な場合は `【要確認: 企業固有情報を補完してください】` を挿入
   - **research_brief.md の軸7-4（志望動機の核）が存在する場合は必ず参照し、特定された「核」を第1文または冒頭段落の中心に置く。複数の強みを列挙する「良い材料の羅列」にしない**
   - 核が定まっていない（research_brief.md が未作成の場合）は `【要確認: 志望動機の中心軸を確認してください】` を挿入
   - 志望動機の文章構造は「核（なぜここか）→ エピソードとの接続（自分の強みとどう合うか）→ 入社後のビジョン」の順で組み立てる
   - **企業の強み・事業を言及する際、具体的な商品名・資産クラス名・施策名を列挙せず、それらが体現する価値・思想に抽象化して表現する。** 固有名詞の列挙は G9（列挙感チェック）に抵触するリスクが高い。
   - アセットマネジメント志望動機で、設問が職種を細かく限定していない場合は、抽象的な運用哲学やESGの大義から入るより、**「経済・企業を分析し投資判断を行う仕事そのものへの関心」→「研究・インターンとの接続」→「その会社で学べる業務範囲や育成構造」→「将来像」** の順で組み立てる
   - 総合コース等で配属が広い場合は、いきなり狭い職種志望に固定せず、幅広い業務理解を経て将来の目標職種へ進みたい流れを許容する
   - 証券リテール志望動機では、**業界構造の優位性を先に語るより、「顧客にどんな価値を返せるか」から入る。** フィービジネスや銀証連携は「顧客本位の提案を支える仕組み」として位置づける
   - 初期配属がリテール前提の証券会社では、いきなり投資銀行・リサーチ等の遠い将来像を主役にしない。まずリテールで顧客理解と提案力を磨く流れを中心に据え、その先の専門性は補助線として書く
   - **志望動機（業界・会社）設問では、エピソードの詰め込みを禁止。** 根拠として使うエピソードは1〜2つに絞り、論理構造の明確さを優先する。バスケット・部活・高校の体験など「遠いエピソード」は原則省く
   - **「なぜこの企業か」の設問では、企業の構造的説明（○○だから→○○になる）より、企業が実際にやっていること・取り組みの姿勢への共鳴を軸にする。** research_brief.md の軸7-4（核心固有性）がある場合は、その企業が「どう行動しているか」を具体的に参照して共鳴を表現する
4. 自己PR・強みは、冒頭1文で強みを平易に言い切る
   - ゼミ・研究の専門性はそのまま用語で押し出さず、「不確実でも判断材料を組み立てられる」「結果だけでなく前提や過程を説明できる」など、仕事での再現可能な力に翻訳してから具体例を置く
   - 設問が純粋な自己PR・強み説明である場合、末尾を機械的な「貴社で貢献したい」で締めない。強みの再定義、価値を出したい方向、仕事観の一文で閉じる
   - 20〜100字程度の短文設問では、抽象名詞を重ねたスローガン調を避け、**一読で意味が入る素直な語句**に言い換える
   - 個性や特徴を複数問われる場合、3つとも「構造化」「設計」など同系統に寄せず、分析力・実行力・対人対応力のように読み手が違いを判別できる粒度で分散させる
   - **自己PRで使用禁止の抽象フレーム語**（文系若手面接官に伝わらないためNG）:

     | NG語彙 | 言い換え先 |
     |--------|----------|
     | 「構造化する」 | 「整理する」「手順に落とす」 |
     | 「再現可能な仕組み」 | 「誰でも使える形」「次の人も同じようにできる形」 |
     | 「設計力の核」「〇〇の核」 | 削除。具体的な行動・成果で体現する |
     | 「品質への執着」「〇〇への執着」 | 削除。「〇〇を確認して〇〇を担保した」と行動で示す |
     | 「誠実な仕組み」 | 「安心して使える形」「問題が起きても確認できる形」 |
     | 「ブラックボックス化」 | 「担当者が退職して手順が不明になった」等の具体的状況で語る |
5. **趣味特技**設問は、本人の人柄・キャラクターを伝えることを最優先にする
   - 強みや業務との接続は、自然に導ける場合のみ1文添える程度にとどめる
   - 「○○の姿勢につながっています」「○○と重なります」など、無理なスキル連結は禁止
   - 実際の趣味・特技をそのまま素直に書く（投資・データ分析など「業務に近い趣味」も自然な表現でOK）
   - 「やりたいことビジョン」設問では、明確な時系列ロードマップ（3〜5年後・10年後など）の作成を原則禁止する。**「現在地からの学びの姿勢 ＋ 最終的に目指す姿」の構成を優先する**。総合職や配属が広い場合は、最初から狭い職種志望に固定せず、幅広い業務理解の上で将来を語る流れにする
6. 学業・ゼミ・研究設問は、卒論・研究が**完了しているか・進行中か**を必ず判定してから書く
   - 完了済み（成果が確定している）: 「〜に取り組み、〜を明らかにしました」の完了形で具体的な成果・苦労・学びを書く
   - **進行中（卒論執筆中・分析途上）**: 完了形で成果を断定しない。以下の構成を使う:
     - 問題意識（既存の限界・課題感）→ 自分が採るアプローチの方向性 → 設計の考え方 → 挑戦したいこと
     - 「〜しました」ではなく「〜を目指しています」「〜に取り組みたいと考えています」を使う
     - 「研究を通じて〇〇の力を養いました」という成果の断定も避け、「この研究で〇〇を身につけていきたい」と姿勢で語る
   - 判定が難しい場合はG3のルールに従い現在進行形を使う

6. ガクチカ・業務改善系エピソードは、**問題の診断 → 確認論点の整理 → 並行実行 → 途中検証 → 成果 → 学び** の順で、読み手がプロセスを追える形にする
   - 「自動化した」「仕組み化した」だけで済ませず、途中でどの検証や差異確認を入れたかを1文で示す
   - 設問が経験説明に留まる場合、最後に企業貢献の一文を機械的に足さない
   - 「挑戦」「困難対応」など価値観系設問では、単なる分析課題よりも、**制約・認識ずれ・納期圧力の中で優先順位をつけて前進させた経験**を優先する
   - 関係者との確認や意図のすり合わせが重要だった案件では、「何を先に確認し、なぜ見切り発車しなかったか」を明示する
   - **成果の結論は数値に限定しない。** エピソードの核心が「再現性・仕組みづくり・持続可能性（誰が担当しても同じ品質を再現できる）」の場合は、それを結論に置く。数値（作業時間の短縮等）はその根拠・例示として添える位置づけにする
6. 既存回答がある設問はそれを優先し、必要に応じて加筆・修正のみ行う
7. 技術的内容が含まれる場合は **G8** に従い汎用化する（`references/guardrails.md` 参照）

**→ Step 4完了後、`references/guardrails.md` を読み込んでG1〜G11の自己チェックを実施する**

## Step 9: 自動レビュー（保存・コミット後に必ず実行する）

保存・コミット完了後、ユーザーへの確認なしで以下のレビューを自動実行する。

### 9-1. 機械チェック

```bash
python /Users/ishidatomonori/Desktop/knowledge/tools/checks/es_checker.py \
  "/Users/ishidatomonori/Desktop/knowledge/移行後ES/<企業名>.md"
```

ERRORが出た場合は即座にユーザーへ提示し、修正を促す（レビューを続けることは可能）。

### 9-2. 設問適合チェック（最優先）

`question-fit-reviewer` エージェントの手順に従い、各設問への回答性を評価する。

- 各設問の「問いの核心」を1行で抽出する
- 回答が設問に正面から答えているか評価する（◎/○/△/✗）
- ズレがある設問はタイプを分類して修正方針を提示する

### 9-3. 可読性チェック

`readability-reviewer` エージェントの手順に従い、文系若手面接官目線で評価する。

- 結論の遅さ・技術用語・長文・抽象語過多を確認する
- 問題箇所と改善案を列挙する

### 9-4. 企業適合チェック

`company-fit-reviewer` エージェントの手順に従い、企業固有性を評価する。

- `company-info/<企業名>/research_brief.md` が存在する場合のみ実施する
- 軸3・軸7-3・軸9-3との照合を行い、企業名置換テストを実施する
- 「どこでも言えること」に留まっている箇所を抽出する

### 9-5. 設問間重複チェック（設問が3問以上の場合のみ）

`consistency-overlap-reviewer` エージェントの手順に従い、横断チェックを実施する。

- 使用エピソードと強みの重複マトリクスを作成する
- 同じエピソード・同じ切り口の重複箇所を抽出する

### 9-6. レビュー結果サマリーの出力

以下の形式で全レビュー結果をまとめて出力する:

```
## 📋 自動レビュー結果: <企業名>

### 機械チェック
- ERROR: X件 / WARNING: Y件
- [問題があれば内容を記載]

### 設問適合 [◎/○/△/✗]
- [設問ごとの評価と指摘]
```
