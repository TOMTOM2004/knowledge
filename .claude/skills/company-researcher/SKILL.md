# Company Researcher スキル

ユーザーから企業名（とコース名）を受け取り、社内ナレッジ + Web調査でリサーチブリーフを生成する。
生成したブリーフは `company-info/<企業名>/research_brief.md` に保存し、es-writer スキルが参照できるようにする。

---

## Step 0: 入力確認

ユーザーの依頼から以下を特定する。

- **企業名**: 「〇〇の企業調査」「〇〇をリサーチ」などから抽出
- **志望コース・部門**: 依頼文に「〇〇コース」「〇〇部門」などが含まれていれば抽出する

**志望コース・部門が指定されていない場合は、調査を開始する前にユーザーに確認する。**

```
「志望するコース・部門を教えてください。
（例：リサーチコース、マーケット部門、ITコース など）
コースが未定の場合はその旨をお知らせください。」
```

コースが未定と明示された場合は、軸4（差別化軸）を「全コース横断での企業研究」として作成し、
代わりに各コースの特徴比較を記述する。

## Step 1: 社内ナレッジ収集（並列）

以下を並列で読み込む。

```
Read: /Users/ishidatomonori/Desktop/knowledge/company-info/<企業名>/notes.md
Read: /Users/ishidatomonori/Desktop/knowledge/ES/md/INDEX.md
Read: /Users/ishidatomonori/Desktop/knowledge/ES/components/best_answers.md
```

notes.md が存在しない場合は「`company-info/<企業名>/notes.md` が見つかりません。Web情報のみで調査します」と伝えて続行する。

notes.md から以下を抽出・整理する:

| 抽出対象 | 確認箇所 |
|---------|---------|
| 事業概要・サービス | 企業概要・業務セクション |
| 職種・コース情報 | 職種・コースセクション |
| 求める人材像 | 採用基準・人材像・説明会メモ |
| キャリアパス | キャリアパスセクション |
| 選考情報 | 選考スケジュール・選考フロー |

## Step 3: Web調査（並列）

社内ナレッジで不足している情報をWebで補完する。
以下の観点ごとにWebSearch / WebFetch を使い、最新情報を取得する。

### 3-1. 調査クエリ設計

| 観点 | 推奨クエリ（例） |
|------|----------------|
| 事業・ビジネスモデル | `<企業名> 事業内容 ビジネスモデル 2024` |
| 収益構造の中身 | `<企業名> 収益構造 信託報酬 収益源 チャネル別` |
| 顧客基盤の内訳 | `<企業名> 顧客 リテール 機関投資家 年金 AUM内訳` |
| 志望コース・部門業務 | `<企業名> <コース名> 業務内容 仕事内容` |
| 求める人材像 | `<企業名> 求める人材 採用 新卒` |
| 競合差別化・業界ポジション | `<企業名> 強み 競合比較 業界` |
| 主要競合他社の特定 | `<業界名> 主要企業 ランキング 比較 2024` |
| 競合との業績・規模比較 | `<企業名> vs <競合名> AUM 売上 規模 比較` |
| 企業の弱み・課題 | `<企業名> 課題 弱み 今後の方針 中期経営計画` |
| 最重要経営課題の優先順位 | `<企業名> 中期経営計画 重点施策 優先課題 2025 2026` |
| キャリアパス | `<企業名> キャリアパス ローテーション 新卒` |
| 初期配属の実態 | `<企業名> 新卒 配属 1年目 2年目 経験談 口コミ` |
| 社風・働き方の実態 | `<企業名> 社風 働き方 残業 口コミ OpenWork` |
| 面接深掘り論点 | `<企業名> 面接 質問 深掘り 通過 体験記` |

コースや部門が特定されている場合は、クエリにコース名を加える。

### 3-2. 信頼できる情報源の優先順位

1. 公式採用サイト・IR資料・中期経営計画
2. 就活情報サイト（マイナビ・リクナビ・ONE CAREER等）
3. 口コミサイト（OpenWork等）は社風・残業・給与の実態把握に積極活用する
4. ニュース記事（事業動向把握）
5. 選考体験記（マイナビ・ONE CAREER）は面接論点把握に活用する

## Step 4: リサーチブリーフの生成

以下の7軸で構造化してブリーフを生成する。

---

### 軸1: 事業・ビジネスモデル

- 収益源・主要サービス・事業ポートフォリオ
- 注力している成長領域

### 軸1-B: 収益構造の中身

- 主な収益モデル（例：信託報酬・手数料・スプレッド等）と各収益源の比重
- チャネル別・顧客層別の収益構造（リテール・法人・機関投資家・グループ内）
- 残高連動型か取引連動型かなど、収益構造の特性

### 軸1-C: 顧客基盤の内訳と戦略

- 主要顧客層（個人・企業年金・機関投資家・グループ会社等）
- 各顧客層へのアプローチ方法・強みのチャネル
- 顧客基盤の課題（例：デジタル化・新規顧客開拓）

### 軸1-D: 最重要経営課題（優先順位付き）

- 中期経営計画や経営者発言から読み取れる重点施策（1〜4位の優先順位で整理）
- 数値目標（AUM・売上・利益率等）と現状とのギャップ
- 自分が入社後に関われる可能性のある課題領域

### 軸2: 配属部署・コースの具体業務

- 日常業務・担当領域・アウトプット（レポート・提案書など）
- チーム構造・上流下流での役割
- 他コースとの分業ライン
- **アウトプット→業務KPIの連結**：アウトプット名だけでなく「そのアウトプットが誰のどのKPIを動かすか」まで記載する（例：「要件定義書」→「保険金支払リードタイム短縮」）

**【IT系コースの場合、必ず追加調査する項目】**
- グループIT会社（情報子会社）と本体IT部門の役割分担：本体採用で配属される組織はどちらで何をするか
- 「内製化」とは具体的に何を内製化しているのか（新規開発 / 保守 / AI実装 / 要件定義のどこか）
- 保守運用 vs 変革案件の比率感（コアシステム担当 vs DX推進担当で大きく異なる）
- ユーザー部門（業務部門）との接点の強さ：IT人材はビジネス側に近いか、開発側に近いか
- アジャイル開発の全社導入状況 vs 部分導入状況（「一部導入」と「全社標準思想」では大きく異なる）
- AI活用の実態：PoC段階か / 本番定着か / どの業務領域に適用しているか
- AI活用の制約：個人情報・説明可能性・ハルシネーション対策として何をしているか

### 軸3: 求める人材像

- 採用HP・説明会で語られる人材像のキーワード
- 評価されやすいスキル・思考スタイル・価値観
- このコース固有の期待値（全社的なものと区別して記載）

### 軸4: 志望コース vs 他コースの差別化軸

- このコースを選ぶ積極的な理由（他コースにはない特徴）
- 「なぜこのコースか」設問で使える観点

### 軸5: キャリアパス

- 入社後3〜5年のモデルルート
- ローテーション・専門深化のパターン
- 中長期でのキャリアゴール例

### 軸6: 企業の強み・弱み（絶対評価）

**強み（ESで言及できる知識として）**
- 独自の経営資源・技術・ネットワーク
- 業界内でのポジション・実績

**弱み・課題（貢献可能性の観点として）**
- 業界・企業固有の課題
- 中期経営計画や経営者発言から読み取れる改善課題
- 自分が入社後に貢献できる余地（軸8と連動）

### 軸7: 同業他社比較・「なぜこの企業か」の論拠

**目的：** 面接・ESで問われる「同業他社ではなく、なぜ貴社か」に答えられる根拠を構築する。

**7-1. 主要競合他社リスト**

| 企業名 | 特徴（一言） |
|--------|------------|
| ...    | ...        |

（業界内の代表的な2〜4社を列挙する）

**7-2. 競合比較マトリクス（総合）**

| 比較軸 | 志望企業 | 競合A | 競合B | 競合C |
|--------|---------|-------|-------|-------|
| 規模（AUM・売上等） | ... | ... | ... | ... |
| 主力戦略・強みの領域 | ... | ... | ... | ... |
| カルチャー・働き方 | ... | ... | ... | ... |
| 採用規模・キャリアパス | ... | ... | ... | ... |
| 競合にない固有の強み | ... | ... | ... | ... |

※ 不明箇所は【要確認】タグをつける。競合より劣る点も正直に記載する。

**【IT系コースの場合、7-3として追加する】**

**7-3. IT内製化比較マトリクス**（IT・デジタル系コース志望時に追加する）

| 比較軸 | 志望企業 | 競合A | 競合B | 競合C |
|--------|---------|-------|-------|-------|
| IT戦略子会社の有無・位置づけ | ... | ... | ... | ... |
| 内製化への姿勢（中計明示か） | ... | ... | ... | ... |
| AI活用の具体度・実装段階 | ... | ... | ... | ... |
| 本体採用IT人材の役割 | ... | ... | ... | ... |
| 変革寄り vs 保守寄り | ... | ... | ... | ... |

**7-3. 「なぜこの企業か」論拠（3段構成）**

```
① 業界全体に惹かれる理由（なぜこの業界か）
② 同業他社ではなく、この企業を選ぶ決め手
   - 競合にはない固有の強み・特徴
   - 自分のやりたいこと・強みとの接点
③ このコース・部門でないといけない理由（軸4と連動）
```

**【NG パターンの指摘】**
- 「規模が大きい」「安定している」など競合でも当てはまる理由はNGとして明示する
- 競合と区別できない志望理由がある場合、代替の角度を提案する

### 軸8: エピソード適合マッピング

社内ナレッジ（ES/md/）のエピソードリストと、軸3（求める人材像）・軸6（弱み/貢献余地）・軸7（なぜこの企業か）を照合する。

**マッチ評価基準**

| 評価 | 定義 |
|------|------|
| ◎ 強くマッチ | 人材像と高い一致。ESの主軸に使える |
| ○ マッチ | おおむね一致。設問によって使える |
| △ 部分的 | 一部は合致するが、そのままでは不足。補強が必要な点を明記 |
| ✕ ミスマッチ / 要注意 | 人材像と合わない、または逆効果になりうる。使用を推奨しない理由を記載 |

**留意事項**
- 無理にエピソードを当てはめない。◎がゼロでも正直に記載する
- ◎がなければ「現在のエピソード群ではカバーしにくい観点」を明示し、追加できるエピソードの方向性を提案する
- △・✕には必ず「何が足りないか / 何が問題か」を一行で添える

**主武器順位（必ず記載する）：**
マッピングが終わったら、面接で実際に使う順番を1〜4位で明示する。◎が複数ある場合は設問への適合度・インパクト・差別化度で優先順位をつける。

```
■ <コース名>向け 主武器順位（面接で使う順）：
1. 主軸（最初に出す）: <エピソード名> → 理由一行
2. 補強: <エピソード名> → 理由一行
3. 差別化: <エピソード名> → 理由一行
4. テーマ深掘り / 仕事哲学: <エピソード名> → 理由一行
```

### 軸9: 面接対策・詰められ論点・逆質問候補

**9-1. 詰められ論点の想定**

- 「なぜこの業界か」「なぜこの企業か（競合ではなく）」「なぜこのコースか」の3段の深掘り想定Q&Aを生成する
- 志望理由の弱点（競合でも言える点・論理の飛躍箇所）を先に洗い出し、防衛ラインを設定する
- エピソード（ガクチカ・自己PR）への深掘り想定（「他にやり方はなかったのか」「最も苦労した点は」等）

**【IT系コースの場合、9-1に必ず追加する6段反論軸】**
「なぜIT全般か→なぜ事業会社ITか（SIer・コンサルではなく）→なぜこの業界のITか→なぜこの業種（例：損保でなく生保）か→なぜこの企業か→なぜこのコース（例：業務職でなくIT職）か」
の6段で切り分けた想定Q&Aを生成する。この6段が埋まると面接での詰め耐性が格段に上がる。

**9-2. 逆質問候補**

- 業務理解・成長環境・戦略理解・社風の4カテゴリから最低2問ずつ生成する
- 「調べればわかる」質問（HP掲載情報）は使わない
- 相手の発言を引き出しつつ入社意欲をアピールできる質問に限定する

**9-3. 自分のエピソードと企業接続マップ**

- 各◎・○エピソードと「企業の何」が接続するかを一行で記載する
- 面接本番で「このエピソードをなぜここで話すのか」を即答できるレベルまで接続を具体化する

## Step 6: ユーザー確認後に保存

ユーザーが「保存して」「OK」などと承認したら:

1. 以下のパスに保存する:
   ```
   /Users/ishidatomonori/Desktop/knowledge/company-info/<企業名>/research_brief.md
   ```
2. ファイルの冒頭に以下のフロントマターを付与する:
   ```yaml
   ---
   company: <企業名>

   created_at: YYYY-MM-DD
   sources:
     - notes.md
     - <使用したURL>
   ---
   ```
3. 保存完了を報告し、【要補完】が残っている場合はその一覧を再提示する。

## 品質ガードレール

- 未確認の数値・シェア・順位は使わない。使う場合は `【要確認】` タグをつける
- Web情報はURLとともに記録し、情報の出所を明示する
- 弱み・課題は批判的トーンではなく「貢献余地」として中立に記述する
- エピソードのマッチ評価は過度に楽観的にしない

### 自分との接続不足チェックリスト（保存前に必ず確認）

ブリーフを保存する前に、以下の問いにすべて答えられるか確認する。答えられない場合は該当箇所に `【要補完】` を付与する。

| チェック項目 | 確認方法 |
|------------|---------|
| この企業の収益モデルを一言で説明できるか | 軸1-B を参照 |
| 顧客層ごとの課題と自分の関わり方をイメージできるか | 軸1-C を参照 |
| 経営の最優先課題に自分の強みをどう接続するか | 軸1-D と軸8を照合 |
| 競合と比べてこの企業を選ぶ理由が1文で言えるか | 軸7-3 を参照 |
| 社風・働き方の実態（良い面・悪い面）を把握しているか | 軸6（社風セクション）を参照 |
| 面接で「なぜここか」「なぜこのコースか」を詰められた際に即答できるか | 軸9-1 を参照 |
| 逆質問を少なくとも3問用意できているか | 軸9-2 を参照 |
| 各エピソードがこの企業の何と繋がるかを一行で言えるか | 軸9-3 を参照 |
